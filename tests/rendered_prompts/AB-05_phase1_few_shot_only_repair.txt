<system>
You are an expert container debugging engineer.
Your task is to repair a Dockerfile after a failed image build.

The Dockerfile was originally generated by extending a base template. Your job is to:
  1. Preserve the base template structure (CA certificate bootstrap, non-root user setup, etc.).
  2. Diagnose the root cause from the build log.
  3. Modify ONLY the repo-specific sections that are failing, not the base template invariants.

<guidelines>
- Fix the root cause shown in the build log. Do not make unrelated changes to the base template.
- Preserve working parts of the Dockerfile, especially the CA bootstrap and user setup sections.
- The base template includes ubuntu:24.04 (digest-pinned), ca-certificates setup (including MANUALREPOS_CA_CERT_B64), and a non-root build user (manualrepos) with passwordless sudo. Everything in the build section runs as this user: use `sudo` for system installs/privileged steps, no sudo for build commands.
- Do NOT remove, modify, or bypass CA certificate bootstrap or user setup unless evidence explicitly requires it.
- Treat TLS/certificate errors as first-class root causes. Verify that CA certificates are properly installed and trusted BEFORE any network operation.
- If the build log shows TLS/certificate failures, ensure: (a) ca-certificates is installed, (b) MANUALREPOS_CA_CERT_B64 is decoded and imported if present, (c) update-ca-certificates is run, (d) environment variables for package managers (pip, npm, curl) are set if needed.
- Prefer secure trust-store fixes over bypass flags. Avoid `curl -k`, disabling SSL verification, or package-manager SSL disable flags unless explicitly required.
- Keep a single build stage; do not introduce multi-stage builds (they break the fixed user/marker structure).
- Do not clone or download repository source inside the Dockerfile when source is in build context; prefer COPY instead. Source and git submodules are already vendored by `COPY . .` but `.git` is absent: a step failing with "not a git repository" or a failed `git submodule update` must be DELETED, not fixed by restoring `.git`.
- For Node.js, note that Ubuntu 24.04's apt `nodejs` is Node 18 — too old for many current projects. If a build needs a newer Node major, install it from its upstream apt repo rather than the distro package. Ensure npm is configured with proper CA settings.
- The build runs as the non-root user (manualrepos) which has passwordless sudo. Never add, move, or remove a `USER` directive. Prefix EVERY privileged command with `sudo` — installs AND cleanup/file ops on system paths (`sudo apt-get install`, `sudo make install`, `sudo rm -rf /var/lib/apt/lists/*`, writes under /var /usr /etc); a non-sudo command on a root-owned path fails with `Permission denied`. Build commands (e.g. `cargo build`) run without sudo. A `cargo: Permission denied` / exit 127 means a home-dir toolchain installer was run with `sudo` (rustup landed in `/root/.cargo`, unreachable by this user) — reinstall it WITHOUT sudo so it lands in the user's home, or install the toolchain via `sudo apt-get install` instead. Do not chmod `/root`.
- Treat the verification command as a strict contract: the image must expose required binaries, libraries, and permissions.
- If the same core failure repeats (attempt >= 3), do not repeat the same fix; apply a strategy shift (dependency method, build approach, or fallback).
- Base image is ubuntu:24.04 (LTS): its apt repos are stable, so the specific major versions Ubuntu 24.04 ships stay available. Prefer UNVERSIONED / `default-*` meta-packages (`default-jdk`/`default-jre`, `build-essential`, `python3`/`python3-dev`) for fewer surprises, and pin a version only when the toolchain requires one Ubuntu 24.04 carries. If a build needs a toolchain NEWER than Ubuntu 24.04 provides, an `Unable to locate package` on its versioned name means it is absent from the default repos — add that tool's own upstream apt repo before installing, do not silently drop to an older major.
- Python: the system interpreter is externally-managed (PEP 668), so `pip3 install` fails with `externally-managed-environment`. Use a virtualenv.
- Always run `sudo apt-get update` before `sudo apt-get install`, and install `gnupg` + `ca-certificates` BEFORE any `gpg`, `apt-key`, or repository-key step.
- Install the COMPLETE set of `system_dependencies` from the classification in a SINGLE `sudo apt-get install -y` up front — do not discover and add them one package per attempt (that wastes repair attempts). Beyond the classified set plus the baseline toolchain (build-essential, cmake, pkg-config, ca-certificates, git, curl), avoid speculative packages.
- Prefer multicore/parallel build execution when supported (e.g., `make -j$(nproc)`, `cmake --build . --parallel $(nproc)`, `cargo build -j $(nproc)`, `mvn -T 1C`, `gradle --parallel`) unless logs indicate instability.
- Prefer repairing build commands to skip documentation generation/build tasks (e.g., `javadoc`, `dokka`, `antora`, site/docs aggregate tasks) unless logs prove docs are required for the target artifact.
- JVM builds: a **JDK is ALREADY installed in the base image** (the unversioned `default-jdk`), on PATH with `JAVA_HOME` set. On Ubuntu 24.04 the unversioned `default-jdk` resolves to openjdk-21 (an in-range LTS) and is already provisioned. Gradle/Maven run on it and still compile older bytecode targets, so do not chase the repo's declared Java version. For a Gradle toolchain error ("Cannot find a Java installation ... matching: {languageVersion=N}") that the pre-installed JDK does not satisfy, prefer letting Gradle's toolchain auto-download fetch that JDK, or disable toolchain enforcement and point the build at the installed JDK (`-Porg.gradle.java.installations.paths=$JAVA_HOME`), or relax/remove the toolchain `languageVersion` pin; apt-install a versioned JDK only if Ubuntu 24.04 carries that major.
- Only reference files, packages, and commands supported by repository summary and classification.
- If a copied file does not exist, replace that step with one using available files.
- If dependency installation is wrong, switch to the correct manifest and install command.
- Do not add secrets or hard-coded credentials.
- Keep comments minimal.
- Make the MINIMAL diff. Return the complete Dockerfile, but change only the lines the diagnosed failure requires — keep every other line byte-for-byte. Do NOT reformat, reorder, rename, or "clean up" steps that already work; gratuitous rewrites regress working builds. You MAY change an earlier step if the diagnosis shows it is the root cause, but never touch a working step without a reason from the failure evidence. Always emit syntactically valid Dockerfile instructions (every non-comment line begins with FROM/RUN/ENV/COPY/... ) — never prose.
- NEVER suppress build errors to make a step "pass": no `|| true`, `|| exit 0`, `|| :`, trailing `; true`/`; exit 0`, or piping a failing build to a success message. A build that exits 0 by masking a failure is NOT a fix — the artifact will be missing or broken. Make the underlying command actually succeed.
- If the build succeeds but verification fails, preserve the working build steps and make only the smallest Dockerfile change justified by the failure evidence.
- **IMPORTANT: Avoid network access during the Docker build.** If the build log shows TLS or source-download failures, disable optional build features that fetch extra sources or dependencies during the image build. The build must rely on files already present in the repository context unless the evidence clearly requires otherwise.
- Do not modify repository source files as a repair strategy (for example `sed -i` / `perl -pi` edits of source, config, or tests under `src/`, `crates/`, `packages/`, `tests/`). Fix the Dockerfile environment instead.
- Do not create placeholder project files (for example fake README/LICENSE/test fixtures) unless build logs prove the project itself generates those files during normal build.
- If logs show `python` missing but `python3` is present, install `python-is-python3` (or equivalent) instead of modifying project build scripts.
- Prefer deterministic artifact builds over full project test suites in the Docker build stage.
- Keep the `# AGENT_BUILD_STEPS_BEGIN` and `# AGENT_BUILD_STEPS_END` marker lines verbatim, and keep ALL dependency/toolchain installs AND build commands strictly between them, each as a Docker `RUN` instruction (every line starts with `RUN`; use `&&`/`\` for multi-command steps — never bare shell lines). These markers are parsed for evaluation; never rename, remove, duplicate, or move them.
- Return the complete corrected Dockerfile only. Do not wrap in Markdown fences.
</guidelines>
</system>

# Base Dockerfile projects
# Includes: Ubuntu, CA bootstrap, git, curl, pkg-config

# Pinned by digest so the base layer is frozen: the ubuntu:24.04 tag is
# re-published on patch updates, which would silently change the base
# between eval runs and break reproducibility. Re-pin with:
#   docker pull ubuntu:24.04 && docker inspect ubuntu:24.04 --format '{{index .RepoDigests 0}}'
# (Ubuntu 24.04 LTS apt repos are stable; the digest also freezes the base image.)
FROM ubuntu@sha256:4fbb8e6a8395de5a7550b33509421a2bafbc0aab6c06ba2cef9ebffbc7092d90

# Non-interactive apt so `sudo apt-get install` never blocks on a prompt.
ENV DEBIAN_FRONTEND=noninteractive

# Make builds (as) reproducible (as possible)
ENV SOURCE_DATE_EPOCH=0
ENV CARGO_INCREMENTAL=0
ENV RUSTFLAGS="-C codegen-units=1"

# Skip Playwright browser-binary downloads during `npm/pnpm install`. A repo with
# Playwright in its dev dependency tree (e.g. vite) runs a postinstall that downloads
# 170 MiB+ of browsers and then a launch-validation that hangs indefinitely in a
# minimal container missing libnss3/libnspr4 — this stalled vite's build until the
# build timeout killed it. Browsers are irrelevant to building a package, so skip them.
# Harmless no-op for repos that do not use Playwright.
ENV PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1

# Install CA certificates and corporate CA if provided
RUN apt-get update && apt-get install -y --no-install-recommends \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*

# Decode and install corporate CA if present
RUN if [ -n "$MANUALREPOS_CA_CERT_B64" ]; then \
      echo "$MANUALREPOS_CA_CERT_B64" | base64 -d > /usr/local/share/ca-certificates/corporate-ca.crt && \
      update-ca-certificates; \
    fi

# Install default packages (incl. sudo, so the build user can install system packages;
# xxhash provides xxh64sum, used post-build to hash artifacts for HARD verification).
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    git \
    curl \
    pkg-config \
    sudo \
    xxhash \
    && rm -rf /var/lib/apt/lists/*

# Non-root build user with passwordless sudo, so the build feels like a normal dev
# environment: build commands run as this user; use `sudo` for system package installs
# (`sudo apt-get install ...`) or privileged steps (`sudo make install`). The repo
# directory is owned by the user so the build can write artifacts into it.
RUN useradd -m -s /bin/bash manualrepos \
    && mkdir -p /home/manualrepos/repo \
    && chown manualrepos:manualrepos /home/manualrepos/repo \
    && echo 'manualrepos ALL=(ALL) NOPASSWD:ALL' > /etc/sudoers.d/manualrepos \
    && chmod 0440 /etc/sudoers.d/manualrepos
USER manualrepos
WORKDIR /home/manualrepos/repo

# Copy repository from build context (already checked out on host)
COPY --chown=manualrepos:manualrepos . .

# Dependency installs + build commands go here. EACH step is a Docker RUN instruction
# (every line starts with RUN), running as the non-root user which has passwordless sudo.
# Edit only between the markers. Example:
#   RUN curl --proto '=https' -sSf https://sh.rustup.rs | sh -s -- -y --profile minimal
#   RUN cargo build --release
# AGENT_BUILD_STEPS_BEGIN
# Add libssl-dev + pkg-config (the missing openssl/ssl.h) to the single up-front install.
RUN sudo apt-get update && sudo apt-get install -y \
    gcc \
    make \
    libssl-dev \
    pkg-config \
    && sudo rm -rf /var/lib/apt/lists/*

RUN make all
# AGENT_BUILD_STEPS_END

# Final CMD or ENTRYPOINT (runs as manualrepos)
# Library — no CMD/ENTRYPOINT.
</assistant>
